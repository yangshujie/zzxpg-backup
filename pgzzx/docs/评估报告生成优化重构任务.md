## 2. 方向一：占位符映射智能化

> **2026-06-13 重设计**：基于方向三/四完成后的实际体验反馈，方向一从原来"仅在后端加注册表"升级为两个层面：

### 2.1 当前痛点（两条重复路径）

```
                    ┌─ 第一次 ─────────────────────────────────────────┐
                    │ calcFlow/edit.vue 阶段四:                          │
模板选择 ──────────→│   → extractPlaceholdersFromHtml(html)              │
```

核心矛盾：**映射配置在模板创建时不管，到流程里才两边重复配**。

**关键原则**：模板自带映射规则（像 Word 样式表一样），运行时不重新发明。模板里写 `{{OverallScore}}` 就天然知道数据来源是 `AUTO_INDICATOR.overall_score`。

```json
{
  "id": 1,
  "templateName": "标准通信对抗评估报告",
  "htmlContent": "<h1>{{ReportTitle}}</h1>...",
  "placeholders": ["ReportTitle", "ExperimentName", "OverallScore", "CustomField"],
  "defaultMappings": [                                     // ★ 新增
    {"key":"ReportTitle","mappingType":"STATIC_TEXT","mappingValue":"通信对抗综合评估报告"},
    {"key":"OverallScore","mappingType":"AUTO_INDICATOR","mappingValue":"overall_score"}
  ],
  "inferredMappings": [                                   // ★ 新增
    {"key":"ExperimentName","mappingType":"TASK_PROPERTY","mappingValue":"taskName","inferred":true},
    {"key":"CustomField","mappingType":"STATIC_TEXT","mappingValue":"","inferred":false}
  ]
}
```

`defaultMappings` 是模板作者保存的（持久化在 DB），`inferredMappings` 是后端实时根据 `PlaceholderRegistry` 计算出来的（对 `defaultMappings` 中不存在的占位符做补充推断，`inferred: false` 标记为需要人工处理）。

#### 2.3.3 前端 calcFlow/edit.vue + flowRunner.vue：选模板自动填充

```javascript
// 原来（从零推断，需要逐行手动调整）
async function onReportTemplateChange(id) {
  const res = await getTemplate(id)
  const placeholders = extractPlaceholdersFromHtml(res.data.htmlContent)
  mappings = placeholders.map(key => createReportMapping(key))
}

// 改后（直接用模板保存的映射，只需覆写少数值）
async function onReportTemplateChange(id) {
  const res = await getTemplate(id)
  const data = res.data
  // 1. 先用模板保存的 defaultMappings
  const saved = data.defaultMappings || []
  // 2. 后端推断的作为补充（已区分 inferred: true/false）
  const inferred = data.inferredMappings || []

  const placeholders = data.placeholders || extractPlaceholdersFromHtml(data.htmlContent)
  mappings = placeholders.map(key => {
    const match = saved.find(m => m.key === key)
         || inferred.find(m => m.key === key)
    return match || { key, mappingType: 'STATIC_TEXT', mappingValue: '', inferred: false }
  })
  // 用户只需把 🟡 标记的 CustomField 手动填上值即可
}
```

### 2.5 底核技术评估：HTML→DOCX 现状 vs 替代方案

> **此节为技术方案探讨**，尚未纳入实施计划。
> 触发原因：方向三统一了 ReportEngine，但渲染链路 `HTML → Jsoup XHTML → docx4j → DOCX` 仍有固有损耗。
> 核心问题：**能否从根本上换掉"HTML 作为中间格式"这个环节？**

#### 2.5.1 当前链路损耗分析

```
HTML模板({{}}占位符)  →  FreeMarker 渲染  →  HTML 字符串     ← 这一半没问题
                                                    │
        ┌───────────────────────────────────────────┘
        ▼
Jsoup XHTML 清洗        ← 内联样式丢 / <style> 不可靠
docx4j XHTML→DOCX       ← 图片 base64→NPE retry / 复杂表格变形
章节编号后注入           ← 设计时不可见，规则写死在 Java 里
```

每一个环节都可能让用户设计的模板和最终生成的 DOCX 产生偏差。

#### 2.5.2 三种可选路线对比

| 维度 | A. 当前方案<br>HTML+Freemarker+docx4j | B. Poi-TL<br>.docx 原生模板 | C. Apache POI<br>纯 Java 构建 DOCX |
|------|------|------|------|
| **模板格式** | HTML 字符串（存 DB TEXT） | .docx 文件（存 DB BLOB 或 MinIO） | 代码控制（无模板文件） |
| **模板设计工具** | Web 可视化编辑器（manage.vue） | **Microsoft Word 或 WPS** | 写 Java 代码 |
| **占位符语法** | `{{key}}` `{{#each}}` `{{#if}}` | `{{key}}` `{{#each}}` `{{#if}}` ⬅ **和现在完全一样** | 无（代码控制） |
| **DOCX 保真度** | ★☆☆ 依赖 docx4j 转换质量 | ★★★ 原生 DOCX，所见即所得 | ★★★ 完全控制 |
| **浏览器预览** | ★★★ 原生 HTML iframe 即可 | ★★☆ 需转 PDF/HTML 后预览（可用 Aspose 或 LibreOffice） | ★☆☆ 需先跑一遍生成 |
| **新增依赖** | 无 | Poi-TL（~300KB jar） | 无（POI 已有） |
| **学习成本** | 中 | **低**（用 Word 设计模板，语法不变） | 高 |
| **循环/条件** | ✓ FreeMarker | ✓ Poi-TL 原生支持 | ✓ Java 控制 |
| **表格/图片** | docx4j 支持差（base64 NPE） | ✓ 原生支持 | ✓ 编程控制 |
| **后端代码量** | 当前 ~500 行 | 替换后 ~150 行 | ~800 行 |
| **替换风险** | 基线 | **低**——ReportEngine 接口不变，只换内部实现 | 高——完全重构 |

#### 2.5.3 推荐：B（Poi-TL）作为中长期方向

**为什么 Poi-TL 适合本项目：**

1. **占位符语法完全兼容** —— `{{OverallScore}}` `{{#each IndicatorSections}}...{{/each}}` `{{#if condition}}...{{/if}}` 写法不变；模板从"存在 DB 里的 HTML 字符串"变成"存在 DB 里的 .docx 文件"，使用者感知的差异只有模板设计方式

2. **方向三已铺好路** —— `ReportEngine.generate(ReportEngineRequest)` 接口不变，两个上层调用方（ZHPG + ZhpgCalcExecutor）零改动。`ReportEngine` 内部替换 docx4j 环节：
   ```
   // 现在：
   FreeMarker 渲染 → HTML 字符串 → Jsoup XHTML → docx4j → DOCX
   // 换成：
   加载 .docx 模板 → Poi-TL fill data → DOCX
   ```

3. **你的用户用例就是 .docx** —— `评估报告参考/JB1801性能评估-出所评估报告.docx` 是正式军工评估报告，你的目标就是生成这种格式。用 Word 设计模板最天然——分页、页眉页脚、字体、表格边框、图片大小全在 Word 里所见即所得

4. **Apache POI 已存在** —— 项目 `pgzzx/pom.xml` 已声明 `poi.version=4.1.2`，`ruoyi-common-core` 和 `ZhpgCalcExecutor` 均已依赖 `poi-ooxml`。Poi-TL 在此基础上增加一个 300KB 的 jar，不需要引入全新生态

**取舍（需要业务层面确认）：**

| 取舍 | 影响 |
|------|------|
| 模板设计从 Web → Word | `manage.vue` 的可视化构建器不再适用，需改为**上传 .docx + 解析占位符 + 配置映射**的模式。对模板设计者的要求从"会拖拽"变成"会用 Word / WPS" |
| 浏览器实时预览变弱 | HTML 预览（`srcdoc` iframe）不再可用；需通过后端将模板渲染为 PDF 后预览（Aspose 已有），或以 HTML 快照方式降级预览 |
| 模板版本管理更自然 | .docx 本身是文件 → 天然适合做版本管理（方向五直接复用文件版本，不需要另建 version 表） |

**如果确定走 Poi-TL，方向一的实施应拆为两个阶段：**

```
Phase 1a（本次）: 映射智能化（不改模板格式）
  → PlaceholderRegistry + defaultMappings + 前端面板
  → 放在模板设计阶段配好，流程里选模板自动加载映射
  → 不改 HTML→DOCX 链路

Phase 1b（中长期，P2）: 模板格式升级
  → pgzc_report_template 支持存储 .docx BLOB
  → Poi-TL 替换 docx4j XHTML→DOCX 步骤
  → ReportEngine 接口不变，内部实现切换
  → manage.vue 改版为"上传 .docx + 解析占位符 + 配置映射"
```

> **决策点**：Phase 1b 的收益（DOCX 保真度 + 用户用 Word 设计模板）是否大于代价（放弃 Web 可视化编辑器）？这取决于实际用户画像——模板是谁来设计、多久设计一次。如果模板设计频率低、由专业人员设计，Word 方案明显更优。

| Freemarker 转换 | ❌ 无，调用方传入已转换的模板 | ✅ 内置 `toFreemarkerTemplate()` |
| 中文章节编号 | ❌ 无 | ✅ `injectChapterNumbering()` |
| 生成方法 | `generateDocx(name, freeMarkerTemplate, data)` | `generateDocx(name, htmlContent, data)` |
| 对外接口 | 需要调用方先转 Freemarker | 传原始 HTML 即可，内部完成全流程 |

**根本问题**：ZHPG 模块的 ReportEngine 是一个**半成品**，占位符转换被上提到 `ReportTemplateServiceImpl` 里做。而 Executor 的 ReportEngine 是**完整封装**。两者暴露的接口不同，调用方也不同。

### 3.2 目标

- **ruoyi-common-report 中只有一个 ReportEngine**，两个模块都通过相同接口调用 ✅
- ZHPG 模块和 ZhpgCalcExecutor 的 ReportEngine 类**删除** ✅

### 3.3 改进方案

#### 3.3.1 统一 ReportEngine 设计

放在 `ruoyi-common-report`：

```java
public class ReportEngine {

    // 完整流程：验证 → 转换 → 渲染 → 清洗 → 生成 DOCX
    public ReportEngineResult generate(ReportEngineRequest request) {
        // 1. 校验占位符
        List<String> warnings = placeholderService.validateTemplateTokens(request.getHtmlContent());

        // 2. 占位符 → Freemarker
        String freemarker = placeholderService.toFreemarkerTemplate(request.getHtmlContent());

        // 3. Freemarker 渲染
        String html = htmlTemplateRenderer.render(request.getTemplateName(), freemarker, request.getDataModel());

        // 4. 章节编号注入（可选）
        if (request.isEnableChapterNumbering()) {
            html = injectChapterNumbering(html);
        }

        // 5. HTML → XHTML
        String xhtml = htmlNormalizer.toXhtml(html);

        // 6. XHTML → DOCX
        byte[] docxBytes = docxGenerator.htmlToDocx(xhtml);

        return ReportEngineResult.builder()
            .docxBytes(docxBytes)
            .warnings(warnings)
            .renderedHtml(html)
            .build();
    }
}

@Data @Builder
public class ReportEngineRequest {
    private String templateName;        // 模板名称（用于 Freemarker 错误定位）
    private String htmlContent;         // 原始 HTML（含 {{}} 占位符）
    private Map<String, Object> dataModel;  // 渲染数据
    private boolean enableChapterNumbering;  // 是否注入中文章节编号
}

@Data @Builder
public class ReportEngineResult {
    private byte[] docxBytes;           // 生成的 DOCX 字节
    private List<String> warnings;      // 占位符校验警告
    private String renderedHtml;        // 渲染后的 HTML（供预览/调试）
}
```

```java
public void generateForResultAsync(Long evalResultId, EvalReportGenerateRequest request) {
    // 1. 立即创建 EvalReportInstance 记录（renderStatus=PENDING）
    EvalReportInstance instance = createPendingInstance(evalResultId, request);

    // 2. 异步执行生成
    CompletableFuture.runAsync(() -> {
        try {
            // 更新进度: HTML_RENDERING
            updateProgress(instance.getId(), "HTML_RENDERING", "正在渲染模板...");
            String html = renderHtml(template, dataModel);

            // 更新进度: CHART_GENERATING
            updateProgress(instance.getId(), "CHART_GENERATING", "正在生成图表(1/5)...");
            // ... 生成图表 ...

            // 更新进度: DOCX_CONVERTING
            updateProgress(instance.getId(), "DOCX_CONVERTING", "正在转换为DOCX...");
            byte[] docx = reportEngine.generate(req);

            // 更新进度: PDF_CONVERTING
            updateProgress(instance.getId(), "PDF_CONVERTING", "正在转换为PDF...");
            byte[] pdf = convertToPdf(docx);

            // 更新进度: UPLOADING
            updateProgress(instance.getId(), "UPLOADING", "正在上传文件...");
            String wordUrl = uploadToMinio(docx);
            String pdfUrl = uploadToMinio(pdf);

            // 完成
            updateInstanceSuccess(instance.getId(), wordUrl, pdfUrl);
            updateProgress(instance.getId(), "DONE", "报告生成完成");
        } catch (Exception e) {
            updateInstanceFailed(instance.getId(), e.getMessage());
        }
    }, reportGenerationExecutor);
}
```

## 6. 方向二：模板设计器所见即所得增强

### 6.1 当前痛点

- 可视化构建器的编辑效果和最终报告差距大
- `{{#each IndicatorSections}}` 循环块在设计器中只是一个静态卡片，看不出循环展开效果
- `dataTable` 组件类型选了 `indicator_summary_table` 也看不到样例数据
- 不知道占位符最终会渲染成什么样子

### 6.2 目标

- 设计器中可以**加载一份样例数据**，实时预览渲染效果
- 循环块可视化展示
- 模板设计时就能验证占位符映射是否正确

## 7. 方向六：报告配置从流程模板解耦为独立实体

### 7.1 当前痛点

- 报告配置（用哪个模板 + 占位符怎么映射）绑定在流程模板的 `configJson.stages.reportOutput` 里
- 如果 5 个流程模板想用同一套报告配置，要在每个流程模板里重复配一遍
- 修改报告模板后，需要逐个更新所有引用它的流程模板
- 报告配置无法独立管理、搜索、复用

### 7.2 目标

- 报告配置成为独立实体"**报告方案**"（ReportPlan）
- 流程模板只通过 ID 引用报告方案
- 一个报告方案可被多个流程模板复用
- 修改报告方案，所有引用的流程自动生效

### 7.3 改进方案

#### 7.3.1 新增实体：报告方案

```sql
CREATE TABLE pgzc_report_plan (
    id                   BIGSERIAL PRIMARY KEY,
    plan_name            VARCHAR(255) NOT NULL COMMENT '方案名称',
    plan_description     VARCHAR(500) COMMENT '方案描述',
    report_template_id   BIGINT NOT NULL COMMENT 'FK → pgzc_report_template.id',
    placeholder_mappings JSONB COMMENT '占位符映射配置',
    output_formats       VARCHAR(128) DEFAULT 'DOCX,PDF' COMMENT '输出格式',
    enable_auto_generate INT DEFAULT 0 COMMENT '是否计算完成后自动生成',
    del_flag             INT DEFAULT 0,
    create_by            VARCHAR(64),
    update_by            VARCHAR(64),
    create_time          TIMESTAMP,
    update_time          TIMESTAMP
);
```

#### 7.3.2 流程模板与报告方案的关联

**旧结构**：
```json
{ "stages": { "reportOutput": { "config": { "reportTemplateId": 1, "placeholderMappings": [...] } } } }
```

**新结构**：
```json
{ "stages": { "reportOutput": { "reportPlanId": 1 } } }
```

#### 7.3.3 迁移兼容

- 读取流程模板时，如果发现旧格式（有 `reportTemplateId`），自动兼容加载
- 保存流程模板时，如果选择的是已有的报告方案，只存 `reportPlanId`
- 如果用户直接选了模板（不通过报告方案），系统自动创建匿名报告方案

#### 7.3.4 管理界面

```
菜单: 模板库管理 > 报告方案库

┌───────────────────────────────────────────────────┐
│ 报告方案列表                                       │
│ ┌──────────┬──────────┬──────────┬──────────────┐ │
│ │ 方案名称  │ 关联模板  │ 引用次数  │ 操作         │ │
│ ├──────────┼──────────┼──────────┼──────────────┤ │
│ │ 通信对抗标准│ 通用动态版│ 3个流程   │ 编辑|复制|删除│ │
│ │ 评估报告   │          │          │              │ │
│ ├──────────┼──────────┼──────────┼──────────────┤ │
│ │ 效能评估  │ 效能评估 │ 1个流程   │ 编辑|复制|删除│ │
│ │ 标准报告   │ 报告模板  │          │              │ │
│ └──────────┴──────────┴──────────┴──────────────┘ │
└───────────────────────────────────────────────────┘
```

## 8. 整体实施路线图

```
Phase 0: 基础夯实（2-3 天）
├── 方向三: ReportEngine 统一 ✅ 已完成（2026-06-12）
└── 方向一: 占位符映射智能化（后端注册表部分）← 当前下一步

Phase 1: 体验提升（3-4 天）
├── 方向一: 占位符映射智能化（前端交互部分）
├── 方向四: 异步生成 + 进度可见
└── 方向四: 预览与生成分离（HTML 先预览）

Phase 2: 可维护性（3-4 天）
├── 方向五: 模板版本管理
└── 方向五: 引用方变更感知

Phase 3: 体验增强（4-5 天）
├── 方向二: 设计器所见即所得
└── 方向六: 报告方案独立实体
```

### 依赖关系

```
方向三(引擎统一) ──→ 方向四(异步生成) ──→ 方向二(所见即所得)
                  ├─→ 方向一(映射智能化) ──→ 方向六(独立报告方案)
                  └─→ 方向五(版本管理)
```

方向三（引擎统一）是**关键路径上的前置**，因为它改变了核心渲染接口，其他方向的实现都会依赖新接口。**该前置已完成**，后续方向可直接基于 `com.ruoyi.common.report.ReportEngine` 的 `generate()` / `generateFromRenderedHtml()` 接口开发。

### 优先级说明

| 优先级 | 方向 | 理由 |
|--------|------|------|
| **P0** | 方向三：ReportEngine 统一 ✅ | 消除两套代码，是其他优化的代码基础（已完成，遗留项见 §3.6） |
| **P0** | 方向一：映射智能化 | 直接减少操作步骤，体验收益最明显 |
| **P1** | 方向四：异步化+进度 | 解决 300s 白等问题，必须有统一的引擎做基础 |
| **P1** | 方向五：版本管理 | 避免数据丢失，且 DDL 已有基础，实现成本较低 |
| **P2** | 方向二：所见即所得 | 工作量大，但能显著降低模板设计门槛 |
| **P2** | 方向六：独立报告方案 | 架构优化，在映射智能化完成后推进更自然 |

## 9. 风险与注意事项

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| ~~统一 ReportEngine 导致两条生成路径行为差异~~（已落地） | 现网报告格式变化 | 实际采用**直接删除旧引擎**而非 deprecated 渐进迁移；通过保留各自原有行为开关规避（ZHPG 章节编号 off / Executor on），单测已覆盖 DOCX 输出结构。两条业务路径端到端联调仍待人工回归（任务 3.7） |
| 异步生成引入并发问题 | 同一 evalResult 同时触发生成 | 加乐观锁——`(eval_result_id, generation_no)` **复合唯一约束**（当前 DDL 只有 `report_code` 唯一索引，且 `nextGenerationNo()` 用 count+1 推算，存在并发竞态），重复点击返回已有 PENDING 实例 |
| 模板版本管理启用后存量数据兼容 | 初期 current_version_id 为 NULL | 代码做 NULL 安全处理，首次保存模板时自动创建 v1 |
| 占位符自动推断覆盖不完整 | 部分占位符仍需手工配 | 保留手工配置入口，推断值标记 `inferred: true` 可覆盖 |
| HTML 预览不经过 DOCX 转换，效果有差异 | 用户看到 HTML 效果好但 DOCX 可能样式丢失 | 预览页给出提示"HTML 预览仅供参考，最终效果以 DOCX 为准" |
