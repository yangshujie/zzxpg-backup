## 2. 方向一：模板设计器重构 — 在线文档编辑器 + 槽位系统

核心思路：**模板只管排版，数据绑定在流程配置时做**。模板设计者打开编辑器就像在用 Word，不接触任何评估业务概念。

### 2.1 设计原则：三层关注点分离

```
┌─ 模板设计层 ──────────────────────────────────────────────────┐
│  谁用：模板设计者                                               │
│  做什么：调字体、建表格、设页边距、标记哪里放数据                    │
│  不知道什么：AUTO_INDICATOR、taskName、评估结果是怎么算出来的       │
│  工具：在线文档编辑器（类似 Word 的 Web 体验）                      │
│  产物：文档 JSON + 槽位列表                                      │
└────────────────────────────────────────────────────────────────┘
                              ↓
┌─ 数据绑定层 ──────────────────────────────────────────────────┐
│  谁用：流程配置者                                               │
│  做什么：选模板 → 系统自动识别每个槽位的数据来源 → 确认/调整        │
│  90% 的槽位自动匹配（通过命名注册表），用户基本不操作               │
│  产物：模板 ID + 少数覆盖值                                        │
│  在哪：calcFlow/edit.vue 阶段四（设计阶段）、flowRunner.vue（执行）│
└────────────────────────────────────────────────────────────────┘
                              ↓
┌─ 渲染输出层 ──────────────────────────────────────────────────┐
│  ReportEngine: 加载文档 JSON → 遍历槽位 → 从评估结果取数据填入    │
│  → POI 写 DOCX → Aspose 转 PDF → 上传 MinIO                     │
│  排版完全由 POI 代码控制，和用户在编辑器里看到的一致                │
└────────────────────────────────────────────────────────────────┘
```

### 2.2 第一层：模板设计（manage.vue）

#### 编辑器选型

采用 **Tiptap**（基于 ProseMirror 的 Vue3 编辑器框架），替代当前的 HTML 拖拽构建器。

| 能力 | 说明 |
|------|------|
| A4 纸效果 | 编辑区按 A4 尺寸渲染，所见即所得 |
| 字体 / 字号 / 颜色 | 工具栏选（宋体、黑体、小四、四号…） |
| 页边距 / 页眉页脚 | 属性面板设置 |
| 插入表格 | 选行列数，调边框样式，合并单元格 |
| 插入图片 | 上传或选已有图片，调宽高 |
| 分页 | 插入分页符，编辑区显示分页线 |
| **标记槽位** | 选中一个区域 → 右键 → "标记为数据槽位" → 命名 |
| 模板预览 | 加载样例数据渲染为 PDF，弹窗预览 |

#### 槽位：不是占位符，是文档中的命名区域

用户编辑时看到的是正常排版好的内容，只有标记过的数据槽位在编辑态有淡色虚线边框：

```
┌──────────────────────┬────────────────────────────┐
│ 试验名称              │ ┌ 槽位: 试验名称 ─────┐    │
│                      │ │                     │    │
├──────────────────────┼─┴──────────────────────┴────┤
│ 试验时间              │ ┌ 槽位: 试验时间 ─────┐    │
│                      │ │                     │    │
└──────────────────────┴─┴──────────────────────┴────┘
```

- 槽位标记只在编辑时显示，导出 DOCX 时不出现
- 槽位名字是中文的（"试验名称"、"综合得分"、"维度表格"…），模板设计者用业务语言命名
- 槽位支持三种类型：文本槽位（填入文本）、表格槽位（填入动态行）、图片槽位（填入图表）

#### 模板存储格式

```json
{
  "document": { /* Tiptap 输出的 ProseMirror JSON */ },
  "slots": [
    { "id": "s1", "name": "试验名称",  "type": "text",  "anchor": { ... } },
    { "id": "s2", "name": "试验时间",  "type": "text",  "anchor": { ... } },
    { "id": "s3", "name": "综合得分",  "type": "text",  "anchor": { ... } },
    { "id": "s4", "name": "维度表格",  "type": "table", "anchor": { ... } },
    { "id": "s5", "name": "雷达图",    "type": "image", "anchor": { ... } }
  ]
}
```

DB 存储：`pgzc_report_template.document_json TEXT` + `pgzc_report_template.slots_json TEXT`（替代原来的 `html_content TEXT`）。

### 2.3 第二层：数据绑定（calcFlow/edit.vue 阶段四）

模板设计者不碰数据绑定。**数据绑定在流程配置时做**，而且大部分是自动的。

#### 槽位命名注册表

系统内置一套槽位名 → 数据源的映射规则（放 `ruoyi-common-report`）：

| 槽位名包含（模糊匹配） | 数据源 | 类型 |
|---|---|---|
| 试验名称、任务名称 | `TASK.taskName` | 文本 |
| 试验时间、开始时间 | `TASK.startTime` | 文本 |
| 评估对象 | `TASK.evaluateTarget` | 文本 |
| 指标体系 | `TASK.indicatorSystemName` | 文本 |
| 综合得分、总分 | `RESULT.score` | 文本 |
| 评定等级、等级 | `RESULT.grade` | 文本 |
| 评估结论 | `RESULT.conclusion` | 文本 |
| 维度表格、维度汇总 | `RESULT.dimensionRows` | 动态表格 |
| 指标明细、底层指标 | `RESULT.indicatorRows` | 动态表格 |
| 雷达图、能力图 | `RESULT.radarChart` | 图片 |
| 关键发现 | `RESULT.keyFindings` | 文本列表 |
| 改进建议 | `RESULT.suggestions` | 文本列表 |

### 2.4 第三层：渲染输出（后端 ReportEngine）

方向三统一的 `ReportEngine` 接口不变，内部实现增加一条新路径：

```java
public ReportEngineResult generate(ReportEngineRequest request) {
    if (request.isJsonTemplate()) {
        // 新路径：Tiptap JSON → POI 直出 DOCX
        //   (1) 解析 document.json → 构建 POI XWPFDocument
        //   (2) 遍历 slots → 从 dataModel 取数据填入
        //   (3) 不经过 HTML / docx4j
        return generateFromJsonTemplate(request);
    } else {
        // 兼容旧路径：HTML+Freemarker+docx4j（存量模板）
        return generateFromHtmlTemplate(request);
    }
}
```

**新路径的关键组件**：

| 类 | 作用 |
|----|------|
| `TiptapDocxBuilder` | 读取 Tiptap JSON → 用 POI 构建 XWPFDocument（段落、表格、图片、页面设置） |
| `SlotDataResolver` | 遍历槽位列表 → 查注册表 → 从 dataModel 取值 → 填入对应 POI 对象 |
| `SlotRegistry` | 槽位命名规则 → 数据源映射表（支持模糊匹配 + 优先级） |

### 2.5 关键技术决策

| 决策点 | 选择 | 理由 |
|--------|------|------|
| 编辑器框架 | Tiptap (ProseMirror) | Vue3 生态最成熟，输出结构化 JSON 而非 HTML |
| DOCX 生成 | Apache POI 直写 | 绕过 HTML→docx4j 的样式丢失问题，排版精确 |
| 槽位标记方式 | 编辑器内置自定义 node/mark | Tiptap 支持插件扩展，槽位渲染为淡色虚线框 |
| 槽位语法 | 不保留 `{{}}` 占位符 | 用户用编辑器右键菜单标记槽位，不手写任何标记符号 |
| 模板预览 | 后端渲染 → PDF → iframe | Aspose 已有，PDF 预览与最终输出完全一致 |
| 存量兼容 | 旧 HTML 模板继续走 Freemarker+docx4j | ReportEngine 同时支持两条路径，旧模板不强制迁移 |

## 8. 风险与注意事项

| 风险 | 缓解 |
|------|------|
| Tiptap → POI 渲染保真度 | Tiptap 输出结构化 JSON，POI 逐元素构建。对每个文档元素（段落/表格/图片/页边距）单独开发映射规则，保持约 95% 以上的还原率 |
| 旧模板迁移 | 新旧格式共存，ReportEngine 根据 `templateFormat` 字段路由到不同路径。旧模板不强制迁移 |
| 槽位命名注册表覆盖不够 | 支持模板级的自定义槽位名称。未命中注册表的槽位在流程配置时标 🟡 提醒手动绑定 |
| 在线编辑器学习成本 | Tiptap 的工具栏和交互模式接近 Word/Notion，配合 A4 纸效果，用户直观感受就是"在浏览器里编辑文档" |
