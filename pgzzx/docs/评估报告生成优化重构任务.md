### 当前存在的 8 个主要问题

| # | 问题 | 影响 |
|---|------|------|
| 1 | **两套 ReportEngine 不一致** — ZHPG 27行 vs ZhpgCalcExecutor 100行 | 相同模板两条路径渲染结果可能不同 |
| 2 | **占位符映射全靠手工配置** — 流程模板编辑和执行两处都要逐行配 | 操作繁琐，易出错 |

| 6 | **报告配置绑定流程模板** — 修改模板后需逐个更新所有引用它的流程模板 | 维护成本高 |
| 7 | **模板设计器无法预览真实数据** — 编辑效果和最终渲染差距大 | 设计效率低 |
| 8 | **Aspose 许可证两处硬编码** — 相同序列号重复出现 | 维护隐患 |

## 2. 方向一：占位符映射智能化（减少手工配置）

### 2.1 当前痛点

用户在「流程模板编辑 — 阶段四」和「flowRunner — Step 4」两处都需要：
1. 选择一个报告模板
2. 对模板里的每个 `{{占位符}}` 逐行选映射类型（`STATIC_TEXT` / `AUTO_INDICATOR` / `TASK_PROPERTY`）
3. 再手工选 mappingValue

一个标准模板有 20+ 个占位符，每行要在下拉框里从 11 个 AUTO_INDICATOR 选项中找到对应的那个。两次重复操作。

### 2.2 目标

- **90% 的占位符自动推断映射**，用户基本不需要手工改
- 只有自定义占位符（模板作者自己起的变量名）才需要人工选择

### 2.3 数据流现状

```
模板 HTML ({{ReportTitle}} {{score}} {{IndicatorTree}} ...)
  │
  ├─ [calcFlow/edit.vue] 阶段四: extractPlaceholdersFromHtml(html) → 占位符列表
  │   └─ reportConfig.placeholderMappings = [{key, mappingType, mappingValue}, ...]
  │       保存到 calcFlowTemplate.configJson
  │
  └─ [flowRunner.vue] Step 4: 从 calcFlowTemplate.configJson 加载 mappings
      └─ 用户调整 → buildReportFields() → resolveReportField() → 调用后端生成
```

### 2.4 改进方案

```
模板 HTML ({{ReportTitle}} {{score}} {{CustomVar}} ...)
  │
  ├─ [Step 1] 提取占位符
  │
  ├─ [Step 2] 对每个占位符查 registerPlaceholder(key)
  │   ├─ 命中内置注册表 → 自动填充 mappingType + mappingValue
  │   └─ 未命中（自定义）→ 默认 STATIC_TEXT，等待人工选择
  │
  └─ [Step 3] 仅展示"未自动推断"的占位符让用户确认
```

## 3. 方向三：两套 ReportEngine 统一

### 3.1 当前痛点

| 对比项 | ZHPG ReportEngine (27行) | ZhpgCalcExecutor ReportEngine (100行) |
|--------|--------------------------|--------------------------------------|
| 占位符校验 | ❌ 无，调用方自行处理 | ✅ `validateTemplateTokens()` |
| Freemarker 转换 | ❌ 无，调用方传入已转换的模板 | ✅ 内置 `toFreemarkerTemplate()` |
| 中文章节编号 | ❌ 无 | ✅ `injectChapterNumbering()` |
| 生成方法 | `generateDocx(name, freeMarkerTemplate, data)` | `generateDocx(name, htmlContent, data)` |
| 对外接口 | 需要调用方先转 Freemarker | 传原始 HTML 即可，内部完成全流程 |

**根本问题**：ZHPG 模块的 ReportEngine 是一个**半成品**，占位符转换被上提到 `ReportTemplateServiceImpl` 里做。而 Executor 的 ReportEngine 是**完整封装**。两者暴露的接口不同，调用方也不同。

### 3.2 目标

- **ruoyi-common-report 中只有一个 ReportEngine**，两个模块都通过相同接口调用
- ZHPG 模块和 ZhpgCalcExecutor 的 ReportEngine 类**删除**

### 3.3 改进方案

#### 3.3.1 统一 ReportEngine 设计

放在 `ruoyi-common-report`：

```java
public class ReportEngine {

    // 完整流程：验证 → 转换 → 渲染 → 清洗 → 生成 DOCX
    public ReportEngineResult generate(ReportEngineRequest request) {
        // 1. 校验占位符
        List<String> warnings = placeholderService.validateTemplateTokens(request.getHtmlContent());

        // 2. 占位符 → Freemarker
        String freemarker = placeholderService.toFreemarkerTemplate(request.getHtmlContent());

        // 3. Freemarker 渲染
        String html = htmlTemplateRenderer.render(request.getTemplateName(), freemarker, request.getDataModel());

        // 4. 章节编号注入（可选）
        if (request.isEnableChapterNumbering()) {
            html = injectChapterNumbering(html);
        }

        // 5. HTML → XHTML
        String xhtml = htmlNormalizer.toXhtml(html);

        // 6. XHTML → DOCX
        byte[] docxBytes = docxGenerator.htmlToDocx(xhtml);

        return ReportEngineResult.builder()
            .docxBytes(docxBytes)
            .warnings(warnings)
            .renderedHtml(html)
            .build();
    }
}

@Data @Builder
public class ReportEngineRequest {
    private String templateName;        // 模板名称（用于 Freemarker 错误定位）
    private String htmlContent;         // 原始 HTML（含 {{}} 占位符）
    private Map<String, Object> dataModel;  // 渲染数据
    private boolean enableChapterNumbering;  // 是否注入中文章节编号
}

@Data @Builder
public class ReportEngineResult {
    private byte[] docxBytes;           // 生成的 DOCX 字节
    private List<String> warnings;      // 占位符校验警告
    private String renderedHtml;        // 渲染后的 HTML（供预览/调试）
}
```

## 4. 方向四：报告生成进度可见 + 异步化

### 4.1 当前痛点

- 前端 `generateEvalReport` 接口 timeout 设为 300,000ms（5 分钟）
- 用户点击"生成报告"后，页面卡住白等，没有任何进度反馈
- 后端是同步执行全流程（渲染HTML → 生成图表 → 转DOCX → 转PDF → 上传MinIO）
- 失败后只有一个 `error_message`，不知道哪一步挂了

### 4.2 目标

- 点击"生成报告"后**立即返回**，不阻塞页面
- 前端能看到**分步骤进度**
- 生成完成有**通知**
- 失败可**重试**

### 4.3 改进方案

#### 4.3.1 后端：异步生成 + 进度状态

**EvalReportInstance 增加进度字段**（DDL 变更）：

```sql
ALTER TABLE pgzc_eval_report_instance ADD COLUMN render_progress VARCHAR(50)
  DEFAULT 'PENDING' COMMENT '生成进度: PENDING/HTML_RENDERING/CHART_GENERATING/DOCX_CONVERTING/PDF_CONVERTING/UPLOADING/DONE';

ALTER TABLE pgzc_eval_report_instance ADD COLUMN render_progress_detail VARCHAR(500)
  COMMENT '进度详情，如"正在生成雷达图(3/5)"';
```

**生成接口改为异步**：

```java
// EvalReportInstanceServiceImpl
public void generateForResultAsync(Long evalResultId, EvalReportGenerateRequest request) {
    // 1. 立即创建 EvalReportInstance 记录（renderStatus=PENDING）
    EvalReportInstance instance = createPendingInstance(evalResultId, request);

    // 2. 异步执行生成
    CompletableFuture.runAsync(() -> {
        try {
            // 更新进度: HTML_RENDERING
            updateProgress(instance.getId(), "HTML_RENDERING", "正在渲染模板...");
            String html = renderHtml(template, dataModel);

            // 更新进度: CHART_GENERATING
            updateProgress(instance.getId(), "CHART_GENERATING", "正在生成图表(1/5)...");
            // ... 生成图表 ...

            // 更新进度: DOCX_CONVERTING
            updateProgress(instance.getId(), "DOCX_CONVERTING", "正在转换为DOCX...");
            byte[] docx = reportEngine.generate(req);

            // 更新进度: PDF_CONVERTING
            updateProgress(instance.getId(), "PDF_CONVERTING", "正在转换为PDF...");
            byte[] pdf = convertToPdf(docx);

            // 更新进度: UPLOADING
            updateProgress(instance.getId(), "UPLOADING", "正在上传文件...");
            String wordUrl = uploadToMinio(docx);
            String pdfUrl = uploadToMinio(pdf);

            // 完成
            updateInstanceSuccess(instance.getId(), wordUrl, pdfUrl);
            updateProgress(instance.getId(), "DONE", "报告生成完成");
        } catch (Exception e) {
            updateInstanceFailed(instance.getId(), e.getMessage());
        }
    }, reportGenerationExecutor);
}
```

```java

```

## 5. 方向五：模板版本管理 + 兼容性保障

### 5.1 当前痛点

- 数据库 DDL 中定义了 `pgzc_template_version` 表和 `current_version_id` 字段，但 **Java 代码完全没启用**
- 修改模板直接覆盖 `html_content`，历史记录全部丢失
- 已生成的报告实例指向的模板 ID，但模板内容已变化，无法用原模板复现
- 流程模板里存的占位符映射，模板修改后可能因占位符增删而失效

### 5.2 目标

- 模板每次保存内容时**自动创建版本记录**
- 报告实例记录**生成时使用的模板版本号**
- 能查看模板的**版本历史和差异对比**
- 模板修改后，引用它的流程模板能**感知变化并提示**

### 5.3 改进方案

#### 5.3.1 数据库已有基础设施

    template_id   BIGINT NOT NULL,
    version_no    VARCHAR(64) NOT NULL,  -- v1, v2, v3...
    html_content  TEXT NOT NULL,
    dsl_json      TEXT,                  -- 占位符结构化定义

#### 5.3.2 版本管理逻辑

**保存模板内容时自动递增版本**：

```java
// ReportTemplateServiceImpl.updateTemplateContent()
public void updateTemplateContent(Long templateId, String htmlContent, String changeLog) {
    // 1. 查当前模板
    ReportTemplate template = getById(templateId);

    // 2. 提取占位符，生成 dslJson
    List<String> placeholders = placeholderService.extractPlaceholders(htmlContent);
    String dslJson = buildPlaceholderDsl(placeholders);

    // 3. 生成下一个版本号
    String nextVersion = getNextVersionNo(templateId); // 从历史查最大版本号+1

    // 4. 创建版本记录
    TemplateVersion version = new TemplateVersion();
    version.setTemplateId(templateId);
    version.setVersionNo(nextVersion);
    version.setHtmlContent(htmlContent);
    version.setDslJson(dslJson);
    version.setChangeLog(changeLog);
    templateVersionMapper.insert(version);

    // 5. 更新当前模板的 html_content 和 current_version_id
    template.setHtmlContent(htmlContent);
    template.setCurrentVersionId(version.getId());
    updateById(template);
}
```

ALTER TABLE pgzc_eval_report_instance ADD COLUMN template_version_no VARCHAR(64)

## 7. 方向六：报告配置从流程模板解耦为独立实体

### 7.1 当前痛点

- 报告配置（用哪个模板 + 占位符怎么映射）绑定在流程模板的 `configJson.stages.reportOutput` 里
- 如果 5 个流程模板想用同一套报告配置，要在每个流程模板里重复配一遍
- 修改报告模板后，需要逐个更新所有引用它的流程模板
- 报告配置无法独立管理、搜索、复用

### 7.2 目标

- 报告配置成为独立实体"**报告方案**"（ReportPlan）
- 流程模板只通过 ID 引用报告方案
- 一个报告方案可被多个流程模板复用
- 修改报告方案，所有引用的流程自动生效

### 7.3 改进方案

#### 7.3.1 新增实体：报告方案

```sql
CREATE TABLE pgzc_report_plan (
    id                   BIGSERIAL PRIMARY KEY,
    plan_name            VARCHAR(255) NOT NULL COMMENT '方案名称',
    plan_description     VARCHAR(500) COMMENT '方案描述',
    report_template_id   BIGINT NOT NULL COMMENT 'FK → pgzc_report_template.id',
    placeholder_mappings JSONB COMMENT '占位符映射配置',
    output_formats       VARCHAR(128) DEFAULT 'DOCX,PDF' COMMENT '输出格式',
    enable_auto_generate INT DEFAULT 0 COMMENT '是否计算完成后自动生成',
    del_flag             INT DEFAULT 0,
    create_by            VARCHAR(64),
    update_by            VARCHAR(64),
    create_time          TIMESTAMP,
    update_time          TIMESTAMP
);
```

#### 7.3.2 流程模板与报告方案的关联

**旧结构**：
```json
{ "stages": { "reportOutput": { "config": { "reportTemplateId": 1, "placeholderMappings": [...] } } } }
```

**新结构**：
```json
{ "stages": { "reportOutput": { "reportPlanId": 1 } } }
```

#### 7.3.3 迁移兼容

- 读取流程模板时，如果发现旧格式（有 `reportTemplateId`），自动兼容加载
- 保存流程模板时，如果选择的是已有的报告方案，只存 `reportPlanId`
- 如果用户直接选了模板（不通过报告方案），系统自动创建匿名报告方案

## 8. 整体实施路线图

```
Phase 0: 基础夯实（2-3 天）
├── 方向三: ReportEngine 统一 ← 所有后续改动的基础
└── 方向一: 占位符映射智能化（后端注册表部分）

Phase 1: 体验提升（3-4 天）
├── 方向一: 占位符映射智能化（前端交互部分）
├── 方向四: 异步生成 + 进度可见
└── 方向四: 预览与生成分离（HTML 先预览）

Phase 2: 可维护性（3-4 天）
├── 方向五: 模板版本管理
└── 方向五: 引用方变更感知

Phase 3: 体验增强（4-5 天）
├── 方向二: 设计器所见即所得
└── 方向六: 报告方案独立实体
```

### 依赖关系

```
方向三(引擎统一) ──→ 方向四(异步生成) ──→ 方向二(所见即所得)
                  ├─→ 方向一(映射智能化) ──→ 方向六(独立报告方案)
                  └─→ 方向五(版本管理)
```

方向三（引擎统一）是**关键路径上的前置**，因为它改变了核心渲染接口，其他方向的实现都会依赖新接口。

### 优先级说明

| 优先级 | 方向 | 理由 |
|--------|------|------|
| **P0** | 方向三：ReportEngine 统一 | 消除两套代码，是其他优化的代码基础 |
| **P0** | 方向一：映射智能化 | 直接减少操作步骤，体验收益最明显 |
| **P1** | 方向四：异步化+进度 | 解决 300s 白等问题，必须有统一的引擎做基础 |
| **P1** | 方向五：版本管理 | 避免数据丢失，且 DDL 已有基础，实现成本较低 |
| **P2** | 方向二：所见即所得 | 工作量大，但能显著降低模板设计门槛 |
| **P2** | 方向六：独立报告方案 | 架构优化，在映射智能化完成后推进更自然 |

## 9. 风险与注意事项

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| 统一 ReportEngine 导致两条生成路径行为差异 | 现网报告格式变化 | 保留旧 ReportEngine 作为 deprecated，渐进迁移；对比测试两条路径输出 |
| 异步生成引入并发问题 | 同一 evalResult 同时触发生成 | 加乐观锁（`generation_no` 唯一约束），重复点击返回已有 PENDING 实例 |
| 模板版本管理启用后存量数据兼容 | 初期 current_version_id 为 NULL | 代码做 NULL 安全处理，首次保存模板时自动创建 v1 |
| 占位符自动推断覆盖不完整 | 部分占位符仍需手工配 | 保留手工配置入口，推断值标记 `inferred: true` 可覆盖 |
| HTML 预览不经过 DOCX 转换，效果有差异 | 用户看到 HTML 效果好但 DOCX 可能样式丢失 | 预览页给出提示"HTML 预览仅供参考，最终效果以 DOCX 为准" |
