# 综合分析计算流程模板设计

## 1. 设计目标

综合分析计算分系统（`ZHFX`）位于评估任务构建、指标体系构建、算法模型配置、评估数据匹配之后，负责把前序配置组织成一条可执行、可复用、可追溯的评估计算链路。

结合当前项目已完成内容与原型设计，本阶段优先落地"计算流程模板"能力：

1. 模板要管理哪些配置，哪些必须项可真正支撑后续计算。
2. 模板如何与现有模块联动，包括指标体系、算法、数据匹配方案、报告模板。
3. 模板发布后如何生成一次计算实例，并保留快照、日志和结果关联。

## 2. 与现有模块的关系

当前仓库中已具备或已明确预留的能力如下：

| 序号 | 模块 | 状态 | 说明 |
|------|------|------|------|
| 1 | 指标体系构建配置（ZBGJ） | 已完成 | 指标、指标模板、指标体系、指标体系模板及算法链配置 |
| 2 | 算法模型配置（SFMX） | 已完成 | 算法管理（含发布/停用状态、参数配置、代码包上传），作为流程模板中的执行算法来源 |
| 3 | 评估数据匹配（PGSJ） | 需求已明确 | 数据匹配方案、映射结果、数据校验（实体和页面尚未落地） |
| 4 | 综合分析计算（ZHFX） | 接口预留 | 后端命名已预留 `/zhpg/calcFlow/**`、`/zhpg/calc/**`，实体和页面尚未落地 |
| 5 | 报告辅助生成（BGSC） | 预留 | 作为流程产物之一，可由流程模板指定输出目标 |

因此，计算流程模板不应重新维护"指标定义""算法定义""数据源定义"，而应承担 **"编排和绑定"** 的职责。

## 3. 设计原则

### 3.1 四阶段固定骨架 + 阶段内可配置

根据原型设计，计算流程模板采用 **4 个固定阶段** 的流水线结构：

```
调度策略及执行模式选配 → 权重计算 → 综合分析计算 → 评估报告模板选配
```

每个阶段内部的具体算法/参数/策略可由用户配置，但阶段顺序固定，不做自由拖拽式流程编排。这样更贴合当前系统成熟度，也便于接口、表结构和权限控制快速落地。

### 3.2 以现有算法能力为主，模型能力预留扩展位

当前仓库已完成的是算法管理（`pgzc_algorithm` + `pgzc_algorithm_param`），独立模型库尚未成型。因此流程模板以"算法引用"为必选能力；"模型引用"作为可选扩展字段预留。

### 3.3 模板配置和运行实例分离

模板描述"应该怎么跑"，运行实例描述"某次任务实际怎么跑、跑到了哪里、产出了什么"，避免模板被运行态信息污染。

## 4. 流程模板四阶段详细设计

### 4.1 阶段一：调度策略及执行模式选配

本阶段定义计算任务的调度方式和数据准备策略，是整条流程的入口。

| 配置项 | 类型 | 必填 | 说明 |
|--------|------|------|------|
| routeStrategy | String | 是 | 路由策略：`FIRST`(第一个) / `LOAD_BALANCE`(负载均衡) / `LAST`(最后一个) / `RANDOM`(随机) |
| execMode | String | 是 | 执行模式：`AUTO`(自动) / `MANUAL`(手动确认) / `STEP_BY_STEP`(逐步执行) |
| dataPlanMode | String | 是 | 数据方案来源：`PGSJ_RESULT`(评估数据匹配结果) / `MANUAL_BIND`(手工绑定) |
| dataPlanId | Long | 条件必填 | 关联评估数据匹配方案ID |
| dataChecks | Array | 否 | 数据校验项：`COMPLETENESS`(完整性) / `LOGIC`(逻辑闭合性) / `CREDIBILITY`(可信性) / `CONSISTENCY`(一致性) |
| allowDataCorrection | Boolean | 否 | 是否允许手工补录或修正数据，默认 false |
| timeoutSeconds | Integer | 否 | 数据准备超时时间（秒），默认 300 |
| retryOnFail | Boolean | 否 | 数据准备失败是否自动重试，默认 false |
| retryTimes | Integer | 否 | 重试次数，默认 0 |

### 4.2 阶段二：权重计算

本阶段负责指标体系中各级指标的权重处理。权重来源可以是指标体系中预配的权重分配算法，也可以在模板中覆盖指定。

| 配置项 | 类型 | 必填 | 说明 |
|--------|------|------|------|
| weightSource | String | 是 | 权重来源：`INDICATOR_SYSTEM`(沿用指标体系配置) / `TEMPLATE_OVERRIDE`(模板覆盖指定) |
| weightAlgorithmId | Long | 条件必填 | 当 weightSource=TEMPLATE_OVERRIDE 时，指定权重计算算法ID（来自算法管理） |
| weightAlgorithmParams | Object | 否 | 权重算法参数覆盖 |
| conductionMode | String | 否 | 权重传导方式：`NONE`(无) / `TOP_DOWN`(自顶向下) / `BOTTOM_UP`(自底向上)，默认沿用指标体系 conductionConfig |
| normalizeMethod | String | 否 | 归一化方法：`MIN_MAX` / `Z_SCORE` / `NONE`，默认 `NONE` |
| weightCheckRule | String | 否 | 权重校验规则：`SUM_TO_ONE`(同层权重和为1) / `WARN_ONLY`(仅警告) / `SKIP`(跳过) |
| failStrategy | String | 否 | 失败策略：`TERMINATE`(终止) / `SKIP`(跳过) / `MANUAL_CONFIRM`(人工确认) |

### 4.3 阶段三：综合分析计算

本阶段是流程核心，执行综合评估算法，产出评估得分和结论。

| 配置项 | 类型 | 必填 | 说明 |
|--------|------|------|------|
| algorithmRefs | Array[Long] | 是 | 综合评估算法ID列表（来自算法管理，可多个串行执行） |
| algorithmChainMode | String | 否 | 算法链执行方式：`SERIAL`(串行) / `PARALLEL`(并行后聚合)，默认 `SERIAL` |
| algorithmParams | Object | 否 | 算法参数覆盖（按算法ID为key） |
| modelRefs | Array[Long] | 否 | 模型引用列表（预留，首版可空） |
| inputKeys | Array[String] | 否 | 输入数据键列表，默认取前序阶段输出 |
| scoreLevelRule | String | 否 | 评分分级规则：`AUTO`(算法自动) / `CUSTOM`(自定义阈值) |
| scoreLevelConfig | Object | 否 | 自定义分级阈值配置，如 `{"EXCELLENT":90,"GOOD":75,"PASS":60}` |
| failStrategy | String | 否 | 失败策略：`TERMINATE` / `MANUAL_CONFIRM` / `USE_FALLBACK`(使用兜底值) |
| timeoutSeconds | Integer | 否 | 计算超时时间（秒），默认 900 |

### 4.4 阶段四：评估报告模板选配

本阶段定义计算结果的输出方式，包括报告生成和结果推送。

| 配置项 | 类型 | 必填 | 说明 |
|--------|------|------|------|
| reportTemplateId | Long | 否 | 关联报告模板ID |
| outputTargets | Array[String] | 否 | 结果输出目标：`RESULT_DB`(结果入库) / `REPORT_BGSC`(推送至报告生成系统) / `ANALYSIS_ZNFX`(推送至智能分析系统) |
| resultFormat | String | 否 | 结果输出格式：`STRUCTURED`(结构化JSON) / `FLAT_TABLE`(扁平表) |
| includeSubScores | Boolean | 否 | 是否包含分项得分明细，默认 true |
| includeConclusion | Boolean | 否 | 是否生成评估结论摘要，默认 true |
| autoArchive | Boolean | 否 | 是否自动归档计算结果，默认 false |

## 5. 业务对象设计

### 5.1 计算流程模板 `CalcFlowTemplate`

主表，对应原型中"流程模板列表"的每一行。

| 字段 | 类型 | 必填 | 说明 |

| calcGranularity | String(50) | 是 | 评估粒度：`EQUIP_EFFECTIVENESS`(装备效能指标评估) / `SYSTEM_TASK`(体系级任务评估) / `PERFORMANCE`(性能指标评估) |

| dataPlanId | Long | 否 | 关联数据方案ID |
| dataPlanName | String(200) | 否 | 冗余：关联数据方案名称（列表展示用） |
| configJson | Text | 是 | 模板配置JSON（含四阶段完整配置） |
| versionNo | String(20) | 是 | 版本号，如 `V1.0` |
| status | String(20) | 是 | `DRAFT`(草稿) / `TESTING`(测试中) / `PUBLISHED`(已启用) / `DISABLED`(已停用) |
| description | String(500) | 否 | 模板说明 |
| del_flag | char(1) | 是 | 删除标志，默认 '0' |

> 继承 `BaseEntity`，自动具备 createBy、createTime、updateBy、updateTime、remark 字段。

## 7. 页面设计

### 7.1 流程模板管理页（列表页）

对应原型上半部分。

**页面标题**：流程模板管理

**页面描述**：管理评估计算流程模板，定义评估任务的计算流程结构。流程模板与评估任务类型、装备类型关联，在流程配置中选用。

**筛选区域**：

| 筛选项 | 控件类型 | 说明 |
|--------|----------|------|
| 适用装备子体系 | 下拉选择 | 如"航天侦察装备子体系"，筛选 equipmentType |
| 评估任务类型 | 下拉选择 | 全部 / 作战试验评估 / 体系级任务评估 / 在役考核评估 / 性能鉴定试验评估 |
| 装备类型 | 下拉选择 | 全部 / 6类装备类型 |
| 新建流程模板 | 按钮 | 打开新建页 |

**表格列**：

| 列名 | 字段 | 宽度 | 说明 |
|------|------|------|------|
| 模板名称 | templateName | 180px | 可点击进入详情 |
| 评估任务类型 | taskType | 130px | 字典翻译展示 |
| 装备类型 | equipmentType | 100px | 字典翻译展示 |
| 评估粒度 | calcGranularity | 130px | 字典翻译展示 |
| 关联指标体系 | indicatorSystemName | 200px | 可点击跳转至指标体系详情 |
| 关联数据 | dataPlanName | 120px | 可点击跳转至数据方案 |
| 状态 | status | 80px | Tag 标签：已启用(success) / 测试中(warning) / 草稿(info) |
| 更新时间 | updateTime | 170px | 格式 yyyy-MM-dd HH:mm |
| 操作 | — | 120px | 按状态展示不同按钮（见下） |

**操作按钮逻辑**（根据原型）：

| 模板状态 | 可用操作 |
|----------|----------|
| 草稿（DRAFT） | 编辑、删除 |
| 测试中（TESTING） | 编辑、启用 |
| 已启用（PUBLISHED） | 编辑、停用 |
| 已停用（DISABLED） | 编辑、启用、删除 |

**四阶段对应的编辑表单**：

**阶段一 - 调度策略及执行模式选配**：
- 路由策略（下拉：第一个/负载均衡/最后一个/随机）
- 执行模式（下拉：自动/手动确认/逐步执行）
- 数据方案来源（下拉：评估数据匹配结果/手工绑定）
- 关联数据方案（下拉选择，联动 PGSJ 已有方案）
- 数据校验项（多选：完整性/逻辑闭合性/可信性/一致性）
- 允许手工补录（开关）
- 超时时间（数字输入，秒）
- 失败重试（开关 + 重试次数）

**阶段二 - 权重计算**：
- 权重来源（单选：沿用指标体系配置/模板覆盖指定）
- 权重算法（当覆盖时展示，下拉选择已发布算法）
- 权重传导方式（下拉：无/自顶向下/自底向上）
- 归一化方法（下拉：min-max/z-score/无）
- 权重校验规则（下拉：同层和为1/仅警告/跳过）
- 失败策略（下拉：终止/跳过/人工确认）

**阶段三 - 综合分析计算**：
- 评估算法（多选/拖拽排序，从已发布算法中选择）
- 算法链执行方式（下拉：串行/并行后聚合）
- 算法参数覆盖（展开各算法参数表单）
- 模型引用（预留，多选）
- 评分分级（单选：算法自动/自定义阈值）
- 自定义分级阈值（当选自定义时展示表格输入）
- 失败策略（下拉）
- 超时时间（数字输入）

**阶段四 - 评估报告模板选配**：
- 报告模板（下拉选择）
- 结果输出目标（多选：结果入库/推送报告系统/推送智能分析）
- 结果格式（下拉：结构化JSON/扁平表）
- 包含分项得分（开关）
- 包含评估结论（开关）
- 自动归档（开关）

## 8. 状态流转设计

### 8.1 模板状态

```
DRAFT ──→ TESTING ──→ PUBLISHED ──→ DISABLED
  ↑          │                        │
  └──────────┘                        │
  (回退至草稿)            DRAFT ←─────┘
                        (复制新版本)
```

规则：
- `DRAFT` → `TESTING`：保存校验通过后可提交测试
- `TESTING` → `PUBLISHED`：联调通过后可正式发布
- `TESTING` → `DRAFT`：允许回退至草稿继续修改
- `PUBLISHED` → `DISABLED`：停用（已存在实例不受影响）
- `DISABLED` → `PUBLISHED`：可重新启用
- 已发布模板不允许直接修改，修改时应走"复制新版本"

### 8.2 计算任务状态

```
WAITING → RUNNING → SUCCESS
                  → FAILED
                  → PAUSED → RUNNING（恢复）
          → TERMINATED
```

## 9. 核心校验规则

### 9.1 模板保存校验

1. 模板名称不能为空且不超过200字符。
2. 必须选择评估任务类型和装备类型。
3. 必须绑定指标体系（indicatorSystemId）。
4. 阶段三"综合分析计算"至少关联一个已发布算法。
5. 路由策略不能为空。

### 9.2 模板发布校验

1. 关联指标体系必须为已发布状态。
2. 关联算法必须为已发布状态。
3. 如关联数据方案，则数据方案必须可用。
4. 模板版本号不能为空。
5. 必须先经过 TESTING 状态。
