> 对应需求：评估任务构建分系统 `PGZC-ZHPG-RWGJ`（《04_试验任务综合评估系统软件需求规格说明 V0.53》）。
> 本文聚焦本期落地范围，并为后续扩展模块预留接口与字段。
> 配套文档：`综合分析计算流程模板设计.md`（ZHFX 计算流程模板，本设计直接引用，不重复实现）。

## 1. 设计目标与范围

### 1.1 背景

RWGJ 分系统由 8 个功能模块组成。系统当前菜单已在「模板库管理」下预留 `任务模板库管理`（`taskTemplateLib`）空占位目录，本设计即填充该占位并打通"任务模板 → 评估任务 → 计算执行"链路。

**核心判断**：RWGJ 所需的大部分底层能力（指标体系、算法模型、计算流程模板、报告模板）系统**已实现**。本期不是从零造，而是在已有资产之上补齐三块真正缺失的内容——**任务模板聚合体、评估任务实例、评估准则**——并以"复用 + 合理跳转"的方式对齐需求技术指标。

### 1.2 本期范围

### 1.4 设计总原则

1. **复用优先，引用不重造**：任务模板只持有对指标体系 / 计算流程模板 / 报告模板 / 评估准则的**引用**，不重新维护其内容。
2. **模板与实例分离**：模板描述"应该怎么配"，评估任务是模板的**实例副本**，编辑任务不回写模板。
3. **跳转闭环**：任务编辑复用模板构建页（`mode=task`），不另造一套配置表单。
4. **统一表 + 类型判别 + JSON**：5 类模板共用一张表，差异落在 `template_type` + `type_config_json`，新增类型不改表结构。
5. **对齐既有枚举**：装备类型、评估类型、指标值类型直接复用系统现有字典值。

## 3. 总体架构

### 3.1 三层模型

```
任务模板(可复用预设)  ──实例化(复制)──▶  评估任务(具体实例·存选择+生命周期)  ──发起──▶  计算任务 CalcTask(运行实例)
pgzc_eval_task_template                  pgzc_eval_task                              pgzc_calc_task
        │                                      │  (id == CalcTask.assess_task_id)
        ├─ indicator_system_id  ──────────────┤   指标体系(ZBGJ)
        ├─ calc_flow_template_id ─────────────┤   计算流程模板(ZHFX，内含算法链/权重/报告)
        ├─ criterion_set_id ──────────────────┤   评估准则(pgzc_eval_criterion)
        └─ report_template_id ────────────────┘   报告模板(BGSC)
```

> 当前 `CalcTask.assess_task_id` 为悬空外键，本设计建立 `pgzc_eval_task` 后即指向它，三层链路闭合。

### 3.2 两个正交分类维度

- **模板类型 `template_type`**（评估方法，决定走哪个构建向导）：
  - 本期：`EQUIP_EFFECTIVENESS`(作战效能) / `SYSTEM_CONTRIBUTION`(体系贡献率) / `TASK_SATISFACTION`(作战任务满足度)
  - 预留：`SPECIAL_CAPABILITY`(专项能力) / `KEY_FACTOR`(关键因素影响分析)
- **模板分类 `classification`**（复用范围，保存时选）：`GENERAL`(通用模板) / `SPECIFIC`(专用模板)
- 二者独立：同一类型下可同时存在通用与专用模板。
- **评估粒度 `calc_granularity`**（沿用现有）：体系级任务评估 / 演习演训任务评估 / 装备效能指标评估 / 性能指标评估。

### 3.3 枚举管理方式（代码管理，不入 sys_dict）

本分系统新增的业务枚举一律**在代码中维护**，**不写入 ruoyi-system 的 `sys_dict` 字典库**，以避免 ZHPG 与 system 服务运行期耦合（与现状一致：`EvalIndicator.indicatorType` 的装备类型映射即以 `@Excel readConverterExp` 在代码内维护）。

- **后端**：ZHPG 模块内 Java 枚举/常量类，建议包 `com.ruoyi.domain.zhpg.enums`（`TemplateTypeEnum / TemplateClassificationEnum / EvalTaskTypeEnum / CriterionRuleTypeEnum / 各 Status 枚举`）。
- **前端**：zzx-sub-release-0422 本地常量映射（建议 `src/api/zhpg/constants.js` 或 `views/zhpg/rwgj/constants.js`），供下拉选项与 label 翻译。
- 不为这些枚举生成 sys_dict 的字典/菜单 SQL。

**枚举清单（code → 中文 label）：**

| 枚举 | 取值 |
|---|---|
| 评估方法类型 template_type | `EQUIP_EFFECTIVENESS`作战效能 / `SYSTEM_CONTRIBUTION`体系贡献率 / `TASK_SATISFACTION`作战任务满足度 / `SPECIAL_CAPABILITY`专项能力(预留) / `KEY_FACTOR`关键因素(预留) |

## 6. 评估准则设计

### 6.1 设计依据

综合需求（自定义/脚本、阈值区间/等级判定/条件组合、最佳值最差值注入）与业界通用做法：
- **JSON 规则引擎**：规则以 `ruleType + conditions + actions + priority` 结构化存储，运行期解释执行，便于免发布更新。
- **指标归一化四型**：极大型(效益型)、极小型(成本型)、中间型、区间型，对应 min-max 与"成本→效益"转换公式。
- **分级阈值"逐级达标"**：等级间设阈值，达到下限方可进阶（优良中差）。

### 6.2 `rule_type` 枚举

| rule_type | 含义 | 本期 |
|---|---|---|
| THRESHOLD | 单阈值判定（≥/≤ 阈值 → 通过/不通过或分值） | ✅ |
| LEVEL_MAP | 等级映射（有序阈值区间 → 等级+分值+结论，逐级达标） | ✅ |
| CONDITION | 多条件组合（IF-THEN，AND/OR 条件组 → 输出） | ✅ |
| FORMULA | 评分公式（按指标值类型归一化 → 标准分） | ✅ |
| SCRIPT | 脚本规则（Python/JS 沙箱表达式） | ⏸ 预留 |

### 6.3 `rule_json` 结构示例

**LEVEL_MAP（等级映射，最常用）：**
```json
{
  "ruleType": "LEVEL_MAP",
  "direction": "BENEFIT",
  "tiers": [
    {"level":"优", "min":90, "score":95, "conclusion":"有效压制"},
    {"level":"良", "min":75, "score":85, "conclusion":"部分压制"},
    {"level":"中", "min":60, "score":70, "conclusion":"部分压制"},
    {"level":"差", "min":0,  "score":40, "conclusion":"未形成压制"}
  ]
}
```

**THRESHOLD：** `{"ruleType":"THRESHOLD","op":">=","threshold":0.85,"passScore":100,"failScore":0}`

**CONDITION（多条件组合）：**
```json
{
  "ruleType":"CONDITION","logic":"AND",
  "conditions":[
    {"indicatorCode":"BER","op":"<=","value":0.001},
    {"indicatorCode":"RESP_DELAY","op":"<=","value":2.0}
  ],
  "then":{"level":"达标","score":100},"else":{"level":"不达标","score":0}
}
```

**FORMULA（归一化，见 6.4）：** `{"ruleType":"FORMULA","method":"MIN_MAX","valueCategory":"成本型"}`

### 6.4 指标归一化公式（FORMULA 用）

| 指标值类型 | 方向 | 归一化公式（结果落 [0,1]） |
|---|---|---|
| 效益型 | 越大越好 | `(x - worst) / (best - worst)` |
| 成本型 | 越小越好 | `(worst - x) / (worst - best)`，或 `min/x` 系 |
| 区间效益型 | 落在 [a,b] 最好 | 区间内=1；区间外按到边界距离衰减 `1 - dist / M` |

> `best/worst` 取 `pgzc_eval_criterion.best_value/worst_value`（缺省回退指标 `value_max/value_min`）。

### 6.5 存储与调用

- **准则集**：多条 `pgzc_eval_criterion` 以 `set_id` 归集为"准则集"；任务模板/评估任务通过 `criterion_set_id` 引用整集。
- **调用**：计算阶段按 `任务ID → criterion_set_id → 按 indicator_id/indicator_level 匹配准则`，识别 `rule_type` 调对应执行器，注入 `best/worst` 完成计算，输出分值/等级/结论。
- 与 ZHFX 阶段三衔接：流程模板阶段三若未配 `scoreLevels`，则回退到任务绑定的评估准则集。

### 6.6 自定义（本期）/ 脚本（预留）

- 自定义 `build_mode=CUSTOM`：图形化配置 THRESHOLD/LEVEL_MAP/CONDITION/FORMULA + 预置公式模板库。
- 脚本 `build_mode=SCRIPT`：`rule_type=SCRIPT`，`rule_json` 存脚本与参数；后端沙箱执行后续接入（本期仅存不执行）。

### 8.4 评估准则（`views/zhpg/rwgj/criterion/`）

- 两栏：左侧装备类型 + 准则列表；右侧准则构建/编辑区。
- 准则库：筛选（装备类型/名称）+ 列表（名称/装备/关联指标层级/构建方式/创建时间/操作 编辑·删除·复制）。
- 新建/编辑：构建方式（自定义/脚本预留）→ 指标层级树选择 → 规则编辑（自定义=阈值/区间/等级/条件 图形化；脚本=代码区）→ 校验预览 → 保存入库。

## 9. 状态流转

- **任务模板**：`DRAFT → TESTING → PUBLISHED → DISABLED`（沿用流程模板规则；已发布不可直接改，走复制新版本）。
- **评估任务**：`CREATED → CONFIGURING → READY → CALCULATING → COMPLETED`（+ PAUSED）。
- **下发状态**：`NONE → SENT → RECEIVED → CONFIRMED → RETURNED`。

---

## 10. 核心校验规则

1. 模板名称非空 ≤200；必选 `template_type`；保存时必选 `classification`。
2. 模板发布校验：关联指标体系/计算流程模板须为可用/已发布状态。
3. 评估任务实例化：来源模板须为 PUBLISHED。
4. 评估准则：`rule_type` 与 `rule_json` 结构一致；LEVEL_MAP 阈值需单调、覆盖全区间；FORMULA 须有 best/worst 或可回退指标 min/max。
5. 删除模板/准则前校验是否被下游任务引用。
